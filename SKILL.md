# Magazine Portfolio Skill

> 这个 skill 是 [guizang-ppt-skill](https://github.com/op7418/guizang-ppt-skill) 的姊妹篇。
> 视觉血缘相同(电子杂志 × 电子墨水)、字体系统相同、主题色继承,但场景不同 ——
> guizang-ppt-skill 是**线下演讲辅助**(横翻 deck,一屏一句话);
> magazine-portfolio-skill 是**招聘方静态阅读**(长滚动,信息密度自洽)。

---

## 这个 Skill 做什么

生成一份**单文件 HTML**的长滚动作品集项目页,视觉基调是:

- **电子杂志 + 电子墨水**混血风格(沿用 guizang 视觉系统)
- **WebGL 流体背景**(只在 Hero 区,正文段克制)
- **衬线大标题(Noto Serif SC + Playfair Display)+ 非衬线正文(Noto Sans SC + Inter)+ 等宽元数据(IBM Plex Mono)**
- **顶部 sticky 章节锚点**(滚到对应章节自动高亮)
- **滚动驱动入场动画**(IntersectionObserver,元素进视口淡入 + 上移)
- **4 章节收口结构**(项目背景 / 设计判断 / 落地交付 / 方法沉淀)
- **段落级信息密度**(每段含主标 + 副标 + 副文 + 锚点金句 + 引用 + 服务视角)

这个 skill 的美学基调跟 guizang-ppt-skill 一致 —— 像 *Monocle* 杂志贴上了代码的样子。
但**场景不一样**:它服务于"招聘方在浏览器里自己扫,30 秒到 3 分钟内 get 你能解决什么问题"。

## 何时使用

**合适的场景**:
- 设计师 / 产品经理 / 独立创作者的求职作品集项目页
- 个人 IP / 自媒体的项目复盘网页
- 需要"招聘方静态阅读 + 自洽 + 信息密度"的项目展示
- 已经过了「**作品集项目梳理协作 skill**」(姊妹 skill)的项目 —— 内容架构稿已经成型

**不合适的场景**:
- 线下分享 / 演讲辅助 → 用 [guizang-ppt-skill](https://github.com/op7418/guizang-ppt-skill)
- 内容还没梳理完,只有零散素材 → 先用「作品集项目梳理协作 skill」做架构稿
- 大段表格数据 / 培训课件 → 不适合(密度仍然偏阅读型,不是数据型)
- 需要多人协作编辑 → 这是静态 HTML

## 工作流

### Step 1 · 需求澄清(动手前必做)

**如果用户已经有完整的「作品页内容架构稿.md」**(4 章节 / 8 段 / 已过质量门),可以跳过直接进 Step 2。

**如果用户只有项目素材或模糊想法**,先去用「作品集项目梳理协作 skill」做架构稿,**不要直接进这个 skill**。

#### 6 问澄清清单(场景从演讲改成招聘方阅读)

| # | 问题 | 为什么要问 |
|---|------|-----------|
| 1 | **这是哪种岗位的作品集?**(设计师 / 产品 / 研究员 / 独立创作者) | 决定术语层 / 服务视角的措辞 |
| 2 | **目标招聘方是谁?**(国内大厂 / 创业公司 / 国际公司 / 自由职业 / 个人客户) | 决定语言风格(中文密度 / 英文术语量 / 形式感) |
| 3 | **这个项目页在作品集里的地位?**(头牌 / 主力 / 补充 / 个人探索) | 决定页数和深度 — 头牌可以 8 段,补充可以 4-5 段 |
| 4 | **有图吗?有几张?在哪?** | 详见下方"图片约定" |
| 5 | **想要哪套主题色?** | 见 `references/themes.md`,5 套预设挑一(默认推荐 🌿 森林墨) |
| 6 | **有没有硬约束?**(隐私 / 客户 NDA / 不能出现 XX 名字) | 避免后期返工打码 |

#### 内容来源(关键)

这个 skill **不替你想内容**,它把内容架构稿里的成品填进 HTML 模板。
所以**进这个 skill 之前,你应该已经有:**

- 4 章节 8 段的命名(项目背景 / 设计判断 / 落地交付 / 方法沉淀;每章 1-3 段,每段一个 4 字标签)
- 每段的:主标 / 副标 / 完整副文(2-3 段)/ 锚点金句 / 1-2 条原话引用 / 服务视角注脚 / 数据卡(如有)/ 图片占位说明
- Hero 区的:项目名 / 副标 / 一段 lead / 数据条(5 个数字)

如果上面任何一项还没有,**先回去做架构稿**。这个 skill 不做内容创作,只做视觉化。

#### 图片约定

- **文件夹位置**:`项目/XXX/images/` 下(和 HTML 同级)
- **命名规范**:`{段号}-{语义}.{ext}`,例如 `1B-contrast.jpg` / `2B-atom.png` / `3B-family-grid-01.png`
- **规格建议**:
  - 单张 ≥ 1600px 宽
  - JPG 用于照片/截图,PNG 用于透明 UI/图表
  - 总大小 < 8MB(招聘方网络条件未知,首屏要快)
- **没图怎么办**:模板支持**虚线占位框**(已内置 `.img-placeholder` 类),先用占位生成结构,等图后期补

### Step 2 · 拷贝模板

```bash
mkdir -p "项目/XXX/images"
cp "<SKILL_ROOT>/assets/template.html" "项目/XXX/index.html"
```

`assets/template.html` 是一个**完整可运行**的长滚动模板:CSS、WebGL Hero 背景、滚动入场动画、顶部 sticky 导航全已预设好。
内容部分留了 8 段占位:Hero + 4 章节封面 + 8 个内容段 + 1 个收尾。

#### 2.1 · 必改占位符(容易漏)

| 位置 | 原始 | 替换为 |
|------|------|--------|
| `<title>` | `[项目名] · [岗位] · [设计师/产品名]` | 实际 deck 标题(如 `AI 火种车 · 品牌全案 · 7 天交付一套品牌系统 · 箱子 设计师作品集`) |
| `<meta name="description">` | `[一句话项目摘要]` | 项目方关键词 + 你的差异化(用于搜索引擎和分享卡片) |
| `.topbar .brand` | `[作者名] · [岗位] 作品集` | 实际作者名(如 `箱子 · 设计师作品集`) |
| `.hero h1` | `[项目名]` | 实际项目名 |
| `.hero .meta` | `[项目分类] · [合作方] · [类型]` | 实际项目元数据 |

每次拷贝完 template.html 第一件事:grep 一下 `[` 确认所有方括号占位符都替换完。

#### 2.2 · 选定主题色(5 套预设 · 不允许自定义)

继承 guizang-ppt-skill 的 5 套主题:

| # | 主题 | 适合 |
|---|------|------|
| 1 | 🖋 墨水经典 | 通用 / 商业 / 不知道选啥的默认 |
| 2 | 🌊 靛蓝瓷 | 科技 / 数据 / AI 产品研究 / 工程师作品集 |
| 3 | 🌿 森林墨 | 公益 / 教育 / 文化 / 可持续 / 自然 |
| 4 | 🍂 牛皮纸 | 怀旧 / 人文 / 文学 / 内容创作 |
| 5 | 🌙 沙丘 | 设计 / 艺术 / 创意 / 视觉品牌 |

**操作**:
1. 基于项目主题推荐一套,或问用户选哪一套
2. 打开 `references/themes.md`,找到对应 `:root` 块
3. **整体替换** `assets/template.html` 开头的 `--ink` / `--ink-rgb` / `--paper` / `--paper-rgb` / `--paper-tint` / `--ink-tint` 这 6 行
4. 其他 CSS 全走 `var(--...)`,无需任何其他改动

**硬规则**:
- 一份作品集**整站统一**主题(所有项目页用同一套色,首页 / 关于页也用同一套)
- 不接受用户给的任意 hex 值 —— 委婉拒绝并展示 5 套让选
- 不混搭(例如 ink 取墨水经典、paper 取沙丘)

### Step 3 · 填充内容

#### 3.0 · 类名预检(最重要)

**这是所有生成问题的源头**。`references/sections.md` 里的段落骨架使用了很多类名,如果 `assets/template.html` 的 `<style>` 里没有对应定义,浏览器会 fallback 到默认样式,排版崩溃。

**在写任何 section 代码之前**:
1. **先 Read `assets/template.html` 整个 `<style>` 块**
2. **对照 sections.md 顶部的 Pre-flight 类名清单**,确认每个用到的类都存在
3. 如果某个类缺失:**在 template.html 的 `<style>` 里补上**,不要在每个 section 里 inline 重写

**容易遗漏的类**(必须预先确认存在):
`.topbar` / `.hero` / `.hero canvas.bg` / `.act-cover` / `.act-num` / `.act-tag` / `.section` / `.section .chrome` / `.section .kicker` / `.section .h-sub` / `.section p.body` / `.section .anchor-quote` / `.section .callout` / `.section .service-note` / `.stats-grid` / `.stat-card` / `.split` / `.compare` / `.pipeline-block` / `.pipeline` / `.step` / `.family-grid` / `.family-cell` / `.img-placeholder` / `.outro` / `.fade-in`

#### 3.1 · 4 章节结构骨架(必须遵守)

每个项目页**严格**按 4 章节结构展开:

| 章节 | 4 字章节名(中文) | 英文副名 | 包含的段 | 段标签例(火种车 LOGO) |
|---|---|---|---|---|
| 一 | **项目背景** | Project Framing | 1A · 1B(2 段) | 项目定位 / 用户分层 |
| 二 | **设计判断** | Design Reasoning | 2A · 2B · 2C(3 段) | 流程压缩 / 视觉分诊 / 两轮迭代 |
| 三 | **落地交付** | Execution & Delivery | 3A · 3B(2 段) | 工艺兜底 / 衍生家族 |
| 四 | **方法沉淀** | Adaptive Methodology | 4A(1 段,作为收尾 outro) | 方法迭代 |

**章节封面**(`.act-cover`):每章节开头一个独立 section,深底反白大字,留白多,作为视觉断点。
**内容段**(`.section`):每段一个独立 section,纸底深字。
**收尾**(`.outro`):章节四直接做成 outro 段,作为整页收口。

**屏号 + 4 字标签**:每段标签都是 4 字(中文),对仗工整。
**反向自检**:每章节问"这一章的 1-3 段能不能合并成 1 段说清楚?"——能合并就合并。

#### 3.2 · 段落级信息密度(必须配齐 6 件)

每段 `<section class="section">` 必须包含:

| # | 字段 | 长度 | 例子 |
|---|---|---|---|
| 1 | `.chrome`(段顶元数据) | 一行 | `1A · 项目定位 / Project Footprint` + `Act I · 02` |
| 2 | `.kicker`(短前缀) | 一行 | `1A · 项目定位` |
| 3 | `<h3>` 主标 | 一句 | `7 天 · 一个人 · 一套品牌全案。` |
| 4 | `.h-sub` 副标 | 一句 | `从接手到交付的硬指标。` |
| 5 | `<p class="body">` 副文 | 2-3 段 | 完整副文,术语 + 用户口吻,见 content-density.md |
| 6 | `.anchor-quote` 锚点金句 | 1 句中英 | `Show, don't tell — 让审议端从「读方案」变成「看方案」。` |
| 7 | `.callout`(脚注层引用) | 1-2 条 | 用户原话 / Get 笔记 / 复盘文档原句 |
| 8 | `.service-note`(服务视角注脚) | 一段 | `招聘方读到的是: ...` |
| 9 | (按需)数据卡 / 对比 / Pipeline / 图片 | 视布局而定 | 见 sections.md |

**反向自检 5 项**(每段写完前必过):
- ✅ **服务视角**:这一段的主语是"我"还是"项目方"?
- ✅ **专业度**:主文里有没有大白话直接出现?
- ✅ **保留口气**:用户的口气词(如"兜底能力")有没有被换成 corporate 术语?
- ✅ **金句锚点**:有没有一句中英对仗术语作为整段的钩子?
- ✅ **素材闭环**:每条主张有 1-3 类佐证支撑吗?

5 项全过 → 这一段可以交付。任意一项不过 → 不要交,改完再过门。

#### 3.3 · 选段落布局

打开 `references/sections.md`,里面有 8 种现成段落骨架:

| 段落布局 | 适用 |
|---|---|
| 1. Hero | 第 1 段(WebGL 背景 + 项目元数据) |
| 2. 章节封面 | 每章节开始 |
| 3. 数据大字报 | 1A 项目定位 · 3A 兜底能力(数据卡) |
| 4. 左文右图 | 1B / 2B(完整副文 + 视觉锚) |
| 5. Before/After 对比 | 2A 流程压缩(双列对比) |
| 6. 双 Pipeline | 2C 两轮迭代(流程拆解) |
| 7. 衍生家族图片网格 | 3B 衍生家族(3×2 图) |
| 8. 收尾 outro | 4A 方法沉淀(深底大引用) |

选对应布局,粘过去,改文案 / 改图片路径即可。

## 核心设计原则(哲学)

> 这些原则是从火种车 LOGO 第一份长滚动作品页迭代出来的。违反任何一条,招聘方阅读体验就垮。

1. **稳定优于性格** — 招聘方需要"安心阅读"的稳定感,不是"性格抓眼球"的冲击感(为什么弃 Maxima 大色块楔形风,选杂志墨水风)
2. **内容自洽优于演讲辅助** — 招聘方一个人扫,看完就 get,不需要任何外部解释。每段含完整副文 + 引用 + 服务视角
3. **结构优于装饰** — 不用阴影、不用浮动卡片、不用 padding box,信息靠**字号 + 字体对比 + 网格留白**
4. **章节呼吸优于段落连贯** — 每个章节封面是深底反白的视觉断点,让招聘方滚到那里就知道"进入新章节"
5. **服务视角优于自我表现** — 每段必须过"服务视角"门:主语是项目方,不是"我"
6. **保留口气词优于 corporate 翻译** — "兜底能力""一眼卯定"这种用户原话保留,不要换成"全链路 / 闭环交付"
7. **术语统一** — 同一概念中英文一致,不混译

## 与 guizang-ppt-skill 的差异速查

| 维度 | guizang-ppt-skill | magazine-portfolio-skill |
|---|---|---|
| 形态 | 横翻 deck | 长滚动单页 |
