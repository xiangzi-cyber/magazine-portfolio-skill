# 梳理阶段工作流(Workflow · 8 阶段 + F2 文案门禁 SOP)

> 当用户**还没有内容架构稿**(只有零散素材或模糊想法)时,Agent 走这套梳理流程,产出能进 HTML 生成的内容架构稿。
>
> 如果用户**已经有完整内容架构稿**(4 章节 8 段已成型),跳过本流程,直接进 SKILL.md 的 Step 2(拷模板)。

---

## 阶段全景

```
[用户给项目名]
    ↓
[阶段 A] 启动判断 — 有无长复盘文?
    ↓
[阶段 B] 7 问轻量复盘  ←→  [阶段 C] 节点接力(并行进行)
    ↓
[阶段 D] 素材打开 + 时间线绑定
    ↓
[阶段 E] 主张拆解 → 进素材采集表
    ↓
[阶段 F] 内容架构稿(8 段)
    ↓
[阶段 F2] 文案稿生成(先保真,再提炼)
    ↓
[阶段 G] 专业包装(词汇升级)
    ↓
[阶段 G2] HTML 内容架构稿(只保留可见网页内容)
    ↓
[阶段 H] 反馈接收 + 直接改
    ↓
[过质量门] → 进入 SKILL.md Step 2(拷模板)
```

---

## 全流程硬规则:阶段结束检查

每完成阶段 A / B / C / D / E / F / F2 / G / H,都要按 `references/rule-gates.md` 的"阶段结束检查协议"做一次收口。

没过检查时,不要继续往下走;先补材料、降级主张、重写文案、修图或重新验证。

---

## 阶段 A · 启动判断

收到用户给的项目名后,先判断:

| 项目状态 | 行为 |
|---|---|
| 已有 5000+ 字长复盘文 | 直接读复盘 → 跳到阶段 D(素材打开) |
| 没有长复盘 | 进阶段 B(7 问轻量复盘) |
| 项目还没做完 | **拒绝** —— 这是事后梳理,不是前期规划 |

---

## 阶段 B · 7 问轻量复盘

**模板**:`references/7-questions.md`

**操作**:
- 复制 `7-questions.md` → 重命名为 `项目入口-XXX.md`(每个项目一份)
- 让用户答 7 题(对话式或填模板都可)

**护栏**:答不出来不硬答,空着 —— **空格本身就是关键信息**。

| Q | 内容 | 用途 |
|---|---|---|
| Q1 | 项目是什么、做给谁 | 定项目语境 |
| Q2 | 项目方一开始最痛的点(原话) | 段 1B 真痛点 |
| Q3 | "普通设计师不会这么做"的关键决定 | 段 2A / 2B / 2C 关键判断 |
| Q4 | 关键转折(分岔口) | 段 2B 视觉锚 |
| Q5 | 最值得给招聘方看的具体物 | Hero + 视觉锚 |
| Q6 | 第三方说了什么(原话) | 引用层 / outro |
| Q7 | 12 字以内副标 | Hero 副标 |

---

## 阶段 C · 节点接力(并行)

用户经常用"节点"思维讲项目过程:**"第一个节点是 X""第二个节点是 Y"**

**Agent 抓节点 + 归位**:
1. 每个节点 → 归到对应的 Q(通常是 Q3 / Q4)
2. **标记用户自己冒出来的关键词**(候选金句 / 候选 skill 概念,如"兜底能力""原子理论")
3. 简短反馈,等用户讲下一节点
4. **用户切换 mode** 时(如"先把素材收集好") → Agent 立即转向素材采集,不继续提问

---

## 阶段 D · 素材打开 + 时间线绑定

用户分享项目素材文件夹后:

1. **列目录**(`ls`):看有哪些文件、什么时间戳
2. **挑关键素材读**(优先级:用户笔记 MD > 关键图像 > 时间锚截图 > PDF/PPT)
3. **每张图描述给用户**:让她相信你看到了
4. **绑时间锚**:文件时间戳 ↔ 节点对应(如 `下载.png 11/7 16:47` = 第一节点的时间锚)
5. **找金句素材**:用户原话(对话中) + 笔记原文 + 复盘文档关键句

**必做输出 1**:**素材时间线表**(列出所有素材的时间顺序 + 对应节点)。

**必做输出 2**:`原文摘录与真实问题拆解-XXX.md`。当素材很多、用户明确批评"假大空 / 没读懂素材 / 不像我",或项目依赖用户个人经历时,必须先写这份文件,再进入阶段 E/F。文件保持短而硬,只写能被源材料支持的内容:

| 栏目 | 写什么 |
|---|---|
| 原文摘录 | 从用户文档 / 对话 / 图片说明中摘关键句,标明来源 |
| 真实问题 | 这组素材里真实发生的生活问题 / 项目问题,不用作品集术语包装 |
| 解决动作 | 用户实际做了什么:提示词、设计、教程、测试、交付、答疑等 |
| 不完美 | AI 不稳定、反馈截图缺失、平台限制、伦理边界、不能商用等 |
| 证据缺口 | 缺原始截图 / 缺链接 / 缺数据时直接标"待补",不能写成强证据 |
| 语气锚点 | 用户自己的词,后续主文要保留,不要替换成空泛术语 |

**门禁**:没有完成这份深读文件时,不要直接写 HTML;已有 HTML 也只能视为草稿。若某条主张只来自 Agent 推断,在深读文件里标"推断",并在作品页中降级或删除。

---

## 阶段 E · 主张拆解 → 进素材采集表

**模板**:`references/material-collection.md`

**操作**:
- 复制 `material-collection.md` → 重命名为 `素材采集表-XXX.md`
- 把作品页要让招聘方相信的每一句话 → 拆成主张
- 每条主张 → 配 **1-3 类佐证**(过程截图 / 产物 / 第三方反馈 / 数据 / 时间锚点)

**反向自检**:**三类佐证都拿不到 → 改主张或删主张**,坚决不写靠嘴说的东西。

---

## 阶段 F · 内容架构稿(8 段)

**模板**:`references/content-architecture.md`

**操作**:
新建 `作品页-XXX-内容架构稿.md`,按 4 章节结构展开:

| 章节 | 4 字章节名 | 包含的段 | 段标签例 |
|---|---|---|---|
| 一 | **项目背景** | 1A · 1B(2 段) | 项目定位 / 用户分层 |
| 二 | **设计判断** | 2A · 2B · 2C(3 段) | 流程压缩 / 视觉分诊 / 两轮迭代 |
| 三 | **落地交付** | 3A · 3B(2 段) | 工艺兜底 / 衍生家族 |
| 四 | **方法沉淀** | 4A(1 段,做成 outro) | 方法迭代 |

**每段含 6 个固定字段**:
1. 屏号 + 4 字标签 + 副标(术语层)
2. 讲清楚一件事(一句话主张)
3. 服务视角(内部检查:这一段让读者看到什么真实能力)
4. 视觉建议(画面构成 + 用哪些素材)
5. 文字建议(主标 / 副标 / 副文 / 锚点金句 / 脚注引用)
6. 素材状态表(✅ / ⏳ / ❌)

详细模板见 `content-architecture.md`。

**反向自检**:每章节问"这一章的 1-3 段能不能合并成 1 段说清楚?" —— 能合并就合并。

---

## 阶段 F2 · 文案稿生成(先保真,再提炼)

内容架构稿通过后,先生成 `作品页-XXX-文案稿.md`,不要直接进 HTML。

**写作顺序**:

1. **先写事实段**:每段先用朴素语言写"真实问题 → 解决动作 → 结果 / 限制",不急着加术语。
2. **再写服务视角**:把"我做了什么"改成"招聘方读到什么能力",但不改变事实主语和责任边界。
3. **再写标题 / 副标 / 金句**:标题从段落事实里长出来,不能先想漂亮标题再倒推故事。
4. **最后做专业包装**:只升级表达层,不升级事实本身;没有证据的地方不能靠术语补强。

**每段固定四行自检**:

| 自检项 | 必须回答 |
|---|---|
| 原文依据 | 这一段来自哪份文档 / 哪张图 / 哪句用户原话? |
| 真实问题 | 用户或项目现场到底卡在哪里? |
| 解决动作 | 用户实际做了什么,不是 Agent 推断了什么? |
| 不写什么 | 哪些缺证据、过度拔高、太像口号的表达要删掉? |

**禁用写法**:

- 不用"赋能 / 打造闭环 / 全链路沉淀 / 多维度验证"这类空泛词补内容。
- 不把"工具能力"写成主角;主角是用户如何解决真实问题。
- 不把缺原始截图的反馈写成强第三方背书。
- 不为了显得专业,删掉用户自己的词、犹豫、限制和不完美。
- 不在文案稿没通过前做 HTML,否则容易产出"漂亮但不像我"的页面。

**文案稿通过标准**:

- 用户不需要看素材也能看懂项目发生了什么。
- 每段至少有一个具体动作或具体证据。
- 读起来像同一个真实的人做的项目,不是 8 段模板化能力标签。
- 不完美被诚实处理,但不变成自我否定。

---

## 阶段 G · 专业包装(词汇升级)

把大白话替换为专业术语,**层次**:

| 层级 | 内容 | 处理 |
|---|---|---|
| **主文层** | 专业术语 + 用户判断口吻 | 大白话必须替换 |
| **锚点金句层** | 1 句中英对仗术语 | 整段的"钩子" |
| **脚注层** | 用户原话引用 | 感性底料,降级不删 |

**通用替换表**(摘要,完整版见 `content-architecture.md`):

| 大白话 | 专业术语 |
|---|---|
| 孩子接受度 | **目标受众分层(Audience Stratification)** |
| 直接出图 | **可视化方案先行(Visual-First Proposal)** |
| 心中的筛子 | **视觉本能识别(Visual Pattern Recognition)** |
| 对下游友好 | **全链路设计交付意识(End-to-End Delivery Awareness)** |
| 反复改 | **迭代审议(Iterative Review)** |

**不要换的词**:
- 用户口气词(如"兜底能力""一眼卯定")—— 必须保留
- 时间锚具体数字(7 天 / 12 小时 / 3cm 等)—— 必须保留

---

## 阶段 G2 · HTML 内容架构稿(终稿口吻隔离)

专业包装通过后,先生成 `作品页-XXX-HTML内容架构稿.md`,不要直接写 HTML。

**目标**:把 MD 中的内部判断转成最终网页可见内容,避免 HTML 出现审稿口吻。

**只保留这些字段**:
- 屏号 / 布局类型 / 章节归属
- 最终标题、副标、正文、引用、图片 caption
- 图片路径、alt、证明的主张
- 是否有重复图片;如重复,写明复用理由

**必须删除或转写**:
- `服务视角`
- `招聘方读到的是`
- `招聘方 get 到的是`
- `内部检查`
- `.service-note` / `.internal-note`

**通过标准**:
- 读 HTML 内容架构稿时,已经像作品页终稿,不是审稿表。
- 每张图片只对应一个最强语义位置;同图复用必须有书面理由。
- 删除所有内部字段后,每屏仍然能读出真实问题、解决动作、证据和价值。

---

## 阶段 H · 反馈接收 + 直接改

**用户反馈类型 → SOP**:

| 反馈关键词 | 行动 |
|---|---|
| "幼稚" / "像小学作文" | 找因果太直白的句子,换成对比张力句式 |
| "大白话" / "不专业" | 应用阶段 G 词汇升级 |
| "方向不对" / "视角错了" | 重新审视主张和服务视角 |
| "顺序不对" | 调整段序,验证时间逻辑 |
| "带节奏" / "GPT-like" | 删 N 选 1 / 删清单 / 改为开放给方向 |

**固定动作**:
1. **简短承认**(2-3 行,指出根源,不长篇道歉)
2. **给一个新方向**(不给 4 个候选)
3. **直接 Edit 文件**
4. **让用户检验**

## 输出质量门(每段写完前过 5 项检查)

| 检查项 | 标准 |
|---|---|
| ✅ **服务视角** | 这一段的主语是"我"还是"项目方"?内部判断是否已转写进正文,且没有泄漏到 HTML? |
| ✅ **专业度** | 主文里有没有大白话直接出现?对照阶段 G 替换表 |
| ✅ **保留口气** | 用户的口气词("兜底能力"等)有没有被换成 corporate 术语? |
| ✅ **金句锚点** | 有没有一句中英对仗术语作为整段钩子? |
| ✅ **素材闭环** | 每条主张有 1-3 类佐证支撑吗? |

**5 项全过 → 这一段可以交付**。

---

## 用户三个自我定位关键词(写作时反复回看)

> 用户在术语库第零部分明确写了:
>
> - **坦诚**
> - **愿意为项目花时间**
> - **愿意打磨新东西**
>
> 凡是与这三个词不一致的措辞(吹方法论 / 吹个人壁垒 / 吹与众不同) → 一律不写。
