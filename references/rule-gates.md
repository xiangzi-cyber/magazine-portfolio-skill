# 规则门禁流程(Rule Gates)

> 用来解决"规则写了但调用时被忽略"的问题。每次使用本 skill 时,先识别任务阶段,只激活当前阶段需要的 2-4 条 P0 规则;不要一次性把全部 checklist 当成工作记忆。

---

## 规则应用架构

本 skill 的规则不再按"注意事项列表"使用,而是按下面结构挂载到流程里。新增规则时,必须补齐这 5 项,否则不要写进 skill。

```text
规则 = 触发条件 + 所属阶段 + 需要读取的文件 + 结束检查 + 失败回退动作
```

| 字段 | 作用 | 示例 |
|---|---|---|
| 触发条件 | 什么时候激活这条规则 | 用户说"进图"、素材很多、开始写 HTML、用户说"假大空" |
| 所属阶段 | 规则挂在哪个流程点 | G1 源材料、G2 写作、G4 Image-only、G6 浏览器验收 |
| 需要读取的文件 | 只加载对应参考,避免规则过载 | `workflow.md` / `image-intake-and-screenshot-proof.md` / `checklist.md` |
| 结束检查 | 阶段结束时必须证明什么 | 已产出文案稿、broken image = 0、落位表已更新 |
| 失败回退动作 | 没过时回哪里修 | 回到源材料、重写文案、撤回文案改动、重新压缩图片 |

**放置原则**:

- 元规则放 `rule-gates.md`,负责"何时检查"。
- 内容 / 写作规则放 `workflow.md`,负责"怎么想清楚"。
- 图片规则放 `image-intake-and-screenshot-proof.md`,负责"怎么进图"。
- HTML / 视觉验收规则放 `checklist.md`,负责"怎么交付前验收"。
- 不要把同一条规则同时散落在 3 个文件里;如果需要重复提醒,在其他文件只写链接或门禁编号。

## 运行循环(Runtime Loop)

每次调用 skill 都按这个循环执行。这个循环比单条规则优先级更高。

```text
1. 识别任务类型 → 2. 选择门禁 → 3. 只读取对应规则包 → 4. 执行阶段 → 5. 阶段结束检查 → 6. 继续 / 回退 / 停止
```

| 步骤 | 动作 | 产物 |
|---|---|---|
| 1. 识别任务类型 | 判断用户是在梳理、写作、进图、HTML、视觉修复还是验收 | 任务类型一句话 |
| 2. 选择门禁 | 从 G0-G7 中选 1-3 个本轮门禁 | `本轮激活门禁: [...]` |
| 3. 读取规则包 | 只读门禁指定文件,避免把全部规则塞进上下文 | 已读取文件列表 |
| 4. 执行阶段 | 按当前阶段产出文件 / 页面 / 清单 | 阶段产物 |
| 5. 阶段结束检查 | 按固定格式写检查结果 | `阶段结束检查` |
| 6. 继续 / 回退 / 停止 | 未通过则回退,通过才进入下一阶段 | 下一步决策 |

---

## 使用方法

每次开始任务,先写一行"本轮激活门禁":

```text
本轮激活门禁: [G1 源材料] [G2 写作] [G5 视觉验收]
```

然后按对应门禁执行。门禁没过时,不要继续进入下一阶段;先补产物或降级主张。

---


## 阶段结束检查协议

每完成一个阶段,必须做一次对应检查,再进入下一阶段。检查可以写进执行记录 / 回复用户 / 当前产物末尾,但不能省略。

固定格式:

```text
阶段结束检查: [阶段名]
- 已产出: [文件 / 页面 / 清单]
- 已检查: [本阶段门禁编号]
- 未通过: [无 / 具体问题]
- 下一步: [继续到哪个阶段 / 回退补什么]
```

如果`未通过`不是“无”,下一步只能是回退补材料、降级主张、修图或重做验证,不能继续推进。

| 写文案 / 改叙事 / 招聘方视角 | G1 + G2 | `workflow.md` | 是否有文案稿,每段是否有事实依据 |
| 加图 / 进图 / 替换图 | G3 + G4 + G6 | `image-intake-and-screenshot-proof.md` | 是否只改图,是否更新落位表,是否验证加载 |

旧规则按编号进入索引,以后不要只写自然语言提醒。

|---|---|---|---|---|
| R-AUTH-01 | 不编造客户、数据、项目方信息 | G1 / G2 | 主张是否有来源 | 降级或删除主张 |
| R-SOURCE-01 | 素材多或用户说假大空时先做源材料深读 | G1 | 是否有`原文摘录与真实问题拆解` | 回到素材阅读 |
| R-WRITE-01 | 先事实段,再服务视角,最后标题 / 金句 | G2 | 文案稿段落顺序 | 重写文案稿 |
| R-WRITE-02 | 保留用户口气词,不全改成术语 | G2 / G7 | 用户原词是否保留 | 回到事实段 |
| R-WRITE-03 | 保义压缩,短判断不能扭曲原意 | G2 / G7 | 事实、角色、因果、结果是否变形 | 恢复原意再压缩 |
| R-IMG-01 | Image-only Mode 只进图不改文案 | G4 | diff 是否只含图片相关改动 | 撤回文案改动 |
| R-IMG-02 | 截图默认做证据板,裸放需用户明确要求 | G3 / G4 | 截图是否有外部承托和短 caption | 重新包装截图 |
| R-IMG-03 | 原图不动,网页图单独压缩命名 | G3 | raw / web 是否分离 | 重新入库 |
| R-HTML-01 | 不发明模板里不存在的类名 | G5 | 类名是否存在于 template CSS | 回到模板类名 |
| R-HTML-02 | 4 章节结构不可随意改 | G5 | act-1 至 act-4 是否完整 | 回到结构稿 |
| R-HTML-03 | 模板激活与品牌系统继承 | G5 / G7 | 是否继承母版 class / token / 节奏,是否没有另起页面骨架 | 回到 template 或已验证母版页起步 |
| R-VIS-01 | 同类组件统一修,不只修截图那一处 | G7 | 是否扫描同类组件 | 扫同类再改 |
| R-VIS-02 | 右侧不能空一大片,版心要一致 | G6 / G7 | 标题宽度、左右栏比例、版心线 | 统一修布局 |
| R-QA-01 | 生成后必须浏览器验收 | G6 | broken image = 0, overflowX = 0 | 回到 HTML / 图片阶段 |

## 新增规则接入模板

以后新增规则,按这个格式写,并放到对应文件。

```text
规则 ID: R-XXX-00
触发条件: 用户说 / 文件状态 / 阶段动作
所属门禁: G?
需要读取: references/xxx.md
阶段结束检查: 必须证明什么
失败回退: 没过时回到哪里
放置位置: rule-gates / workflow / image SOP / checklist
```

如果一条规则无法写出"阶段结束检查"和"失败回退",它暂时不进入 skill,避免继续变成无效提醒。

## 阶段内最小规则集

### 梳理 / 写作任务

激活: G1 + G2。

- 先读材料,再写判断。
- 先事实段,再服务视角,最后标题 / 金句。
- 缺证据的内容只能标"待补 / 文字证据",不能包装成强证明。

### 进图 / 替换图任务

激活: G3 + G4 + G6。

- 进图 = 入库命名 + 压缩处理 + 替换占位 + 验证加载 + 更新清单。
- 不新增标题、段落、方法论或新叙事。
- 截图默认做证据板;用户明确说裸放时才裸放。

### HTML 生成任务

激活: G2 + G5 + G6。

- 没有文案稿不进 HTML。
- 动手前读模板 CSS,不要发明不存在的类名。
- 动手前证明模板已激活:母版起步、class family、字体 token、章节节奏都对得上;若已变成另一套骨架,先迁回母版。
- 生成后必须浏览器验收,不要只做静态检查。

### 视觉修复任务

激活: G6 + G7。

- 用户指出一个问题时,先判断是不是同类组件共性。
- 版心、空白、线框、光感、图文节奏要统一修。
- 视觉修复不顺手改已确认文案。

## 常见失效点

| 失效表现 | 原因 | 修正 |
|---|---|---|
| 写得很用力但空 | 跳过 G1,没有先读出真实问题 | 回到源材料深读 |
| 文案漂亮但不像用户 | 跳过 G2,先写标题 / 方法论 | 回到事实段 |
| 加图时页面被重写 | 跳过 G4 | 切回 Image-only Mode |
| 同一个 skill 但品牌系统没起来 | 跳过模板激活检查,页面另起 class / token / 字体系统 | 回到 template 或已验证母版页起步 |
| 规则太多记不住 | 一次加载全部 checklist | 每次只激活本阶段门禁 |
| 只修了一处截图问题 | 跳过 G7 | 找同类组件统一修 |
