# 质量检查清单(Checklist)

生成完作品页 HTML 之后,**逐项对照**。
4 级分类:**P0 必过**(否则不能交付)/ **P1 强烈建议** / **P2 锦上添花** / **P3 可选优化**。

---

## P0 · 必过(否则不能交付)

### P0-0 · 类名预检(最高优先级)

- [ ] 所有 sections.md 用到的类(`.section` / `.split` / `.compare` / `.pipeline` / `.family-grid` / `.stat-card` / `.callout` / `.anchor-quote` / `.service-note` 等)都在 `assets/template.html` 的 `<style>` 块里有定义
- [ ] 没有发明新类名(用 inline `style="..."` 替代未定义的类)
- [ ] **大标题显示成衬线字体**(显示成非衬线 = 类预检没做,`.h-hero` / `<h3>` 在 template 缺失)

### P0-1 · 4 章节结构

- [ ] 章节顺序:**项目背景 → 设计判断 → 落地交付 → 方法沉淀**(不可调整)
- [ ] 每章节有独立 `.act-cover`(章节四例外,直接做成 `.outro`)
- [ ] `id="act-1"` / `id="act-2"` / `id="act-3"` / `id="act-4"` 都设置了
- [ ] 每段 `.section` 归属正确(不能 1A 段出现在章节二里)

### P0-2 · 段落级信息密度(每段必有)

每段 `<section class="section">` 必须有:
- [ ] `.chrome` 段顶元数据
- [ ] `.kicker` 短前缀
- [ ] `<h3>` 主标
- [ ] `.h-sub` 副标
- [ ] `<p class="body">` 至少 2 段副文
- [ ] **`.service-note` 服务视角注脚**(P0 关键!没有就是没过门)

### P0-3 · 必改占位符

- [ ] `<title>` 替换(不能留 `[项目名]`)
- [ ] `<meta name="description">` 替换
- [ ] `.topbar .brand` 替换
- [ ] `.hero h1` / `.hero .meta` / `.hero .subtitle` / `.hero .lead` 替换
- [ ] **grep `[` 确认所有方括号占位符都替换完**

### P0-4 · 主题色

- [ ] 用了 5 套预设之一(不能自定义 hex)
- [ ] `:root{}` 6 行变量(`--ink` / `--ink-rgb` / `--paper` / `--paper-rgb` / `--paper-tint` / `--ink-tint`)整体替换
- [ ] 没有混搭(ink 取一套、paper 取另一套)

### P0-5 · 服务视角原则

每段 `.service-note` 必须用「招聘方读到的是」或「招聘方 get 到的是」开头。

**反向自检**:这段话主语是"我"还是"项目方"?
- ✅ "招聘方读到的是:这人在 toC 项目第一秒就在做目标受众分层"
- ❌ "我做了目标受众分层,这是品牌设计师才有的能力"

### P0-6 · 不要中英混译

同一术语**全文只用一个版本**:
- ✅ Visual-First Proposal(英文术语) + 中文「可视化方案先行」(出现一次,后续都用一种)
- ❌ "Visual-First Proposal / 可视化方案先行 / 视觉优先方案 / 图片化方案" 混用

### P0-7 · 保留用户口气词

**不要换的词**:
- "兜底能力" → 不要换成"全链路 / 闭环交付"
- "一眼卯定" → 不要换
- "因为我干过下游的工作" → 不要换
- "鹿火车只是元素清单" → 不要换

### P0-8 · 锚点金句格式

每段(除 `.outro`)有 1 句 `.anchor-quote`:
- [ ] 中英对仗
- [ ] < 30 字
- [ ] 不直接重复主标
- [ ] 写在副文之后,作为整段的视觉钩子

### P0-9 · 前后一致性

- [ ] **交付物数量前后一致**:Hero 说几类 → 1A 数据卡写几类 → 落地交付展示几类 → Outro 总结几类,**数字必须对得上**
- [ ] **"我做的"vs"团队做的"明确区分**:作品集只放自己做的核心交付物,别人做的要标注"团队协助""由 XX 完成",不能混为一谈
- [ ] **标签命名一致**:导航栏标签、章节封面标题、段顶 `.chrome` 元数据三者统一,不能前面叫「设计判断」后面叫「设计决策」
- [ ] **时间线无空白**:如果有"X 月 X 日定稿→X 月 X 日交付"的间隔,必须交代中间做了什么

### P0-10 · 视觉 QA:版心、空白、光感

- [ ] **同类组件统一修**:发现一个 `.act-cover` / `.visual-plate` / `.section h3` / `.outro` 失衡时,检查并修所有同类组件,不要只修截图那一处
- [ ] **版心一致**:`.act-cover h2/.act-sub`、下一段 `.section .chrome/.kicker/h3`、`.visual-plate .plate-title` 的左边界应在同一条主版心线上
- [ ] **标题不被压窄**:桌面端 `.section h3` / `.visual-plate .plate-title` / `.outro blockquote` 不应只占页面 40% 以下同时右侧大面积空白;优先放宽 `max-width`
- [ ] **右侧不能空一大片**:如果左侧是大标题/大段文字,右侧没有图片、卡片或方法列表承接,必须改为横跨两栏、放宽文字宽度或补充视觉证据
- [ ] **线框不过量**:分类不等于满格边框;大面积表格线会造成阅读噪音。优先使用轻分隔线、留白、色彩锚点、字号层级
- [ ] **收尾页光感温和**:避免黑底白字超大标题 + 巨型高饱和色块 + 右侧灰字堆叠;优先浅底、轻色条、方法卡片、均衡两栏
- [ ] **桌面/手机都过**:桌面检查横向空白和版心;手机检查标题不溢出、不竖排、不重叠

## P1 · 强烈建议

### P1-1 · 数据卡数字长度

- [ ] `.stat-card .num` **2-3 字符**(不超过 4)
- [ ] 单位用 `<span class="unit">` 包(天 / h / cm / × / K / M)
- [ ] 不用 `font-size:auto` —— 用 vw 单位

### P1-2 · 图片占位规范

- [ ] 每个 `.img-placeholder` 加比例修饰类(`.r-1x1` / `.r-3x2` / `.r-3x4` / `.r-16x9`)
- [ ] `.img-placeholder` 的 `.label` 写明"待补 + 用途 + 比例"(如"对照视觉 · 4:3")
- [ ] 衍生家族 `.family-cell` 6 个 cell 都用 `.placeholder`(方图 1:1)

### P1-3 · 滚动入场动画

- [ ] 大标题 / 副文段 / 数据卡 / 引用块 / 锚点金句 / 图片占位 都加 `.fade-in`
- [ ] **不要给 `<section>` 整体加** —— 应该是内部元素分别淡入

### P1-4 · 顶部导航

- [ ] `.topbar .progress` 包含 5 个锚点(`#hero` / `#act-1` 至 `#act-4`)
- [ ] 每个 `<a>` 有 `data-section="..."` 属性
- [ ] JS 自动给当前章节的 `<a>` 加 `.active` 类(IntersectionObserver 已绑定)

### P1-5 · 响应式

- [ ] < 900px 时:`.topbar .progress` 自动隐藏
- [ ] < 900px 时:所有网格降级成单列
- [ ] < 900px 时:`.hero h1` 字号增大到 18vw(避免太小)

### P1-6 · 章节四是 `.outro` 不是 `.section`

- [ ] 章节四直接做成 `<section class="outro" id="act-4">`
- [ ] 没有 chrome / kicker / h3 / 副文 / 锚点金句 / 服务视角(outro 简化)
- [ ] 包含:kicker + blockquote + epilogue + stamp + tags

### P1-7 · 引用 callout 的 src 来源

- [ ] 每个 `.callout` 必须有 `<span class="src">— [来源]</span>`
- [ ] 来源格式:`— [文档名 · 时间]` 或 `— [人名 · 角色]`

### P1-8 · 内容密度平衡

- [ ] **内容太多?拆板块**:如果一个 section 超过 4 段 body + 3 个 callout,拆成 3B-1 / 3B-2 / 3B-3 多个子板块,每个配独立图片占位符
- [ ] **内容太少?合并板块**:如果一个章节只有 1 页内容,不要单独设章节封面,合并到相邻章节(如「视觉统筹」合并到「项目定位」)
- [ ] **标签数量 ≤ 4 个**:导航栏作品页 + 章节标签最多 4 个,超出的合并归纳
- [ ] **长段落先保义压缩**:把 120 字以上的长正文改成关键点时,先标出事实/角色/动作/结果,确保压缩后不改变用户原意
- [ ] **图文节奏有起伏**:不要把字多图多塞在同一屏;按"文字判断 → 图片证据 → 文字判断 → 图片高潮"拆开

### P1-9 · 数字物料兜底(通用)

如果有表情/小程序/网页等数字交付物:
- [ ] 写出平台规格约束(尺寸/格式/透明底等)
- [ ] 说明从设计稿到上架的中间工序(压缩/审核/格式转换)

## P2 · 锦上添花

### P2-1 · WebGL Hero 性能

- [ ] WebGL canvas 离开视口时暂停渲染(template 已内置 IntersectionObserver)
- [ ] shader 颜色跟主题色协调(默认电子墨水流动 shader 适配 5 套主题)

### P2-2 · stamp 和 tags

- [ ] outro 的 `.stamp` 包含 4-5 个字段(项目 / 类型 / 时间 / 身份 / 归属)
- [ ] outro 的 `.tags` 包含 3-5 个关键词(本项目方法论标签)

### P2-3 · 图片实填(替换占位)

如果有真实图:
- [ ] 文件路径 `images/{段号}-{语义}.{ext}`
- [ ] 替换 `.img-placeholder` 为 `<figure class="frame-img r-XXX"><img src="..." alt="..."></figure>`
- [ ] alt 描述准确(招聘方关 JS 时要有 fallback)
- [ ] 设计作品集默认图文比例约 **6:4**;图片是主叙事材料,不是文字旁边的辅助装饰
- [ ] 图片独立页要有节奏:总览图 / 过程墙 / 细节证据 / 成果归档,不要所有图同密度堆成一堵墙

### P2-4 · 手机预览

- [ ] 在手机浏览器里完整滚一遍,确认布局没断
- [ ] `.split` 和 `.compare` 在手机上变单列
- [ ] 字号在手机上 ≥ 14px(`max(14px, 1.1vw)` 形式)

### P2-5 · 章节封面副文

- [ ] `.act-cover .act-sub` 加了 `**强调**` 文字(用粗体标重点)
- [ ] 副文 < 80 字

## P3 · 可选优化

### P3-1 · OG 图(Open Graph 卡片)

如果分享到微信 / Twitter,需要 OG meta:
- [ ] `<meta property="og:title">` / `<meta property="og:description">`
- [ ] `<meta property="og:image">`(项目主视觉,1200×630)

### P3-2 · 页面加载性能

- [ ] WebGL fallback(老浏览器无 WebGL 支持时,Hero 显示纯色背景)
- [ ] 字体 `display:swap`(避免 FOIT)
- [ ] 图片懒加载(`loading="lazy"`)

### P3-3 · 多语言

- [ ] `<html lang="zh-CN">` 设置正确
- [ ] 如果作品集面向英文招聘方,考虑做一份英文版(但不要中英混排)

### P3-4 · 跳转下一项目

- [ ] `.site-footer` 的"下一个作品"链接到下一个项目页(不是 `#`)
- [ ] 可加 prev / next 双向导航

### P3-5 · 阅读进度

- [ ] 顶部加一个细进度条(显示当前滚动到整页的百分比)
- [ ] 招聘方扫到一半能 get "还有 30% 没读"

## 反向自检 5 项(每段写完前过)

写完每段 `<section>`,**逐条对照**:

- ✅ **服务视角**:这一段的主语是"我"还是"项目方"?有没有 `.service-note`?
- ✅ **专业度**:主文里有没有大白话直接出现?
- ✅ **保留口气**:用户的口气词("兜底能力"等)有没有被换成 corporate 术语?
- ✅ **金句锚点**:有没有 `.anchor-quote` 一句中英对仗?
- ✅ **素材闭环**:每条主张有 1-3 类佐证?

5 项全过 → 这一段可以交付。

## 整页交付前的最终自检

### 版心/空白扫描(建议用浏览器脚本)

交付前至少肉眼或脚本扫一次:

- `.section h3`
- `.visual-plate .plate-title`
- `.act-cover h2`
- `.outro blockquote`
- `.hero .subtitle`

如果某个标题在桌面端宽度 `< 42vw` 且右侧空白很大,检查:
1. 是不是文字本身很短?短章节名可接受
2. 是不是 `max-width` 太小?优先放宽
3. 右侧是否有图片/卡片/方法列表承接?没有就拓宽文字或补视觉内容
4. 下方说明是否只占左栏?必要时横跨两栏

- [ ] **所有 P0 项过完**(不过等于不能交付)
- [ ] **所有占位符替换完**(grep `[` 应为 0)
- [ ] **浏览器预览**(macOS:`open index.html`)
- [ ] **滚一遍确认**:Hero → 章节封面 1 → 段 1A → 段 1B → 章节封面 2 → 段 2A → 2B → 2C → 章节封面 3 → 段 3A → 3B → outro 章节四
- [ ] **顶部导航**点击各个章节,正确跳转
- [ ] **手机预览**一次(布局不能断)
