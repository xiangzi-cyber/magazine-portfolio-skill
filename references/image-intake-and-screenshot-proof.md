# 图片进图与截图证据 SOP

> 当用户说"上图 / 进图 / 放图 / 替换图 / 图片资产 / 图床 / Eagle / 压缩包 / 截图美化 / 证据截图"时使用。

作品集里的图片不是装饰,而是主叙事材料。尤其是设计师作品集,真实证据也要有设计力:真实 ≠ 难看,证据 ≠ 直接截图堆上去。

---

## 0. 核心判断

1. **真实素材优先**:用户提供的 LOGO、KT 板、手举牌、表情包、落地截图、公众号证据、第三方引用,优先级高于生成图。
2. **生成图只做两类事**:补充气质参考、统一场景包装。不要把关键交付物重新画成另一个版本。
3. **证据要被包装**:截图可以真实保留,但需要统一比例、留白、背景、阴影、标题层级,让它像作品集里的证据板。
4. **小闭环优先**:先跑通一个项目 / 一个章节 / 一组素材,再复制流程。不要一开始让用户维护过重的资产系统。

---

## 1. 最小可行进图流程

用户给图后,按下面顺序处理:

1. **解压 / 定位**:确认图片来源与项目归属。
2. **筛选**:挑出能支撑叙事的图,不要平均分配。优先保留最终成果、过程证据、第三方引用。
3. **命名**:按项目、章节、角色命名,让文件名自带落位信息。
4. **处理**:压缩、裁切、打码、透明底清理、色调统一。
5. **入库**:原图不动,网页图单独放。
6. **替换 HTML**:更新图片路径与 alt 文本。
7. **浏览器验证**:检查是否加载、是否模糊、是否裁坏、是否造成版式失衡。

建议用户节奏:一次只给一类图,例如"先火种车 KT 板"、"再表情包截图"。

---

## 2. 资产文件夹建议

完整结构:

```text
portfolio-assets/
  raw/          原图,不动
  edited/       打码、裁切、调色、透明底清理
  plates/       拼成作品集展示板或证据板
  web/          压缩后进入网页的版本
  manifest.json 每张图对应哪个占位
```

如果用户觉得流程太长,用轻量版:

```text
assets/{project}/raw/     原图备份
assets/{project}/web/     HTML 实际引用图
assets/图片素材清单.md     人能读懂的落位表
```

轻量版先跑通,后续再扩展 `edited/` 和 `plates/`。

---

## 3. 命名规则

推荐格式:

```text
{project-prefix}-{section}-{role}-{semantic}[-v2|@2x|final].ext
```

例子:

- `hzc-00-logo-final.png`
- `hzc-00-hero-brand-system-final-v2.png`
- `hzc-3b3-sticker-proof-board.png`
- `hzc-3b3-preview-01-wechat-panel-framed.jpg`
- `csf-2a-proposal-proof-v2.jpg`

当浏览器缓存旧图、用户替换了更清晰版本、或图的含义变化时,不要覆盖旧文件名硬刷新;用 `-v2` / `final-v2` / `@2x` 这种版本名,再更新 HTML 引用。

---

## 4. 截图证据美化

截图类素材常见问题:比例乱、边缘脏、手机截屏太窄、文字太小、直接放上去像聊天记录而不像作品集证据。

处理原则:

1. **保留原截图内容**:不要重画、不要改写、不要让 AI 生成一个"类似截图"替代真实证据。
2. **统一外部包装**:背景、圆角、投影、间距、标题编号、说明文字统一。
3. **按落位反推比例**:网页横向证据板优先 16:9 / 16:10;手机截图可做 2-3 张并列卡片。
4. **截图不可裁掉关键信息**:日期、来源、关键界面状态、标题、反馈文字应可读。

如果安装了 `guizang-ppt-skill`,优先读:

```text
guizang-ppt-skill/references/screenshot-framing.md
```

并使用:

```text
guizang-ppt-skill/assets/screenshot-backgrounds/
```

用它的截图背景图和缩放规则做 CleanShot X 风格包装。只有当截图太长、太窄、信息太杂,需要重构成信息板时,才考虑让图像模型生成"重排版证据板";即便重排,也要保留原截图作为真实性来源。

---

## 5. 品牌系统总览图

品牌总览图应该像一页 Brand Identity System Overview,不是素材拼贴。

推荐流程:

1. 先确定落位:核心识别、KT/KV、手举牌、表情包、路线图/官方引用、价值导航。
2. 用真实素材填充每个模块。
3. 如果用户自己用生图或设计工具做出更好的整板图,优先接纳用户最终图,把它作为作品集 hero / 主图插入。
4. 不要把已经被项目抛弃的物料写成最终交付;例如用户明确说房车应用后来没有了,就不要作为成果展示。

---

## 6. 版式自检

每次上图后检查:

- 图片是不是主叙事,而不是文字的附属插图。
- 同组图片比例、圆角、阴影、边距是否统一。
- 是否出现"左边巨大标题,右边大片空白"的失衡。
- 是否出现"字很多 + 图很多"堆在同一屏,需要拆页或拆段。
- 证据截图是否足够好看,能体现设计师的审美处理能力。
- 图片是否清晰,大屏预览不糊;必要时换更高清源或使用 `@2x`。
