# 内容架构稿模板(Content Architecture)

> 把项目梳理(7 问 + 节点接力 + 素材采集)的产物,组织成"作品页一段讲一件事"的内容架构稿。
>
> 这是 **内容层** —— 不是视觉稿。视觉稿(长滚动 HTML)在内容架构定下来之后再做。

---

## 这个文档的角色

```
[7 问 + 节点接力 + 素材采集]  →  [内容架构稿(本模板填充版)]  →  [长滚动 HTML]
```

内容架构稿 = **填好的本模板** = HTML 生成的输入。

---

## 4 章节硬骨架(必须遵守)

每个项目页 **严格** 按 4 章节结构展开:

| 章节 | 4 字章节名 | 英文副名 | 包含的段 | 段标签例(火种车 LOGO) |
|---|---|---|---|---|
| 一 | **项目背景** | Project Framing | 1A · 1B(2 段) | 项目定位 / 用户分层 |
| 二 | **设计判断** | Design Reasoning | 2A · 2B · 2C(3 段) | 流程压缩 / 视觉分诊 / 两轮迭代 |
| 三 | **落地交付** | Execution & Delivery | 3A · 3B(2 段) | 工艺兜底 / 衍生家族 |
| 四 | **方法沉淀** | Adaptive Methodology | 4A(1 段,做成 outro) | 方法迭代 |

**反向自检**:每章节问"这一章的 1-3 段能不能合并成 1 段说清楚?" —— 能合并就合并。让每段都不可被合并、每章节都不可被拆散。

## 每段「讲清楚一件事」的写法规则

主标(`<h3>`)= **这一段要让招聘方相信的一句话**。

**正确**:
- "7 天 · 一个人 · 一套品牌全案。"(数据 + 关键词 + 节奏)
- "利益相关者 ≠ 终端用户。"(术语 + 对仗)
- "AI 给的是方差分布,设计师的工作是分诊。"(术语 + 类比)

**错误**:
- "我做了一个挺好的项目"(大白话)
- "基于 toC 公益设计场景的目标受众分层结构性识别"(学究堆砌)
- "项目挺成功的,客户也很满意"(空话)

**关键平衡**:**专业术语骨架 + 口语化节奏**。

---

## 服务视角注脚的写法规则

每段必有 `服务视角` 一段。**必须用「招聘方读到的是」开头**。

**正确**:
> 招聘方读到的是:这人在 toC 项目的第一秒就在做「目标受众分层」 —— 不被项目方需求清单牵着走,会先识别真正的服务对象。

**错误**:
> 我做了目标受众分层,这是品牌设计师才有的能力。 (主语是"我",自我表现)

---

## 锚点金句的写法规则

每段(除 outro)必有 1 句 `.anchor-quote`。**中英对仗 + < 30 字 + 不重复主标**。

经典模式:
- "Show, don't tell — 让审议端从「读方案」变成「看方案」"
- "本能先响,逻辑跟上 — 内化到位的方法论,长成视觉本能"
- "Compositional Layering — 把 LOGO 拆成可独立设计、可整合迭代的三层结构"
- "Seed Prompt → Family Tree — 一次方法论投入,多次媒介复用"

## 大白话 → 专业术语 替换表(关键)

| 大白话 | 专业术语 | 来源 |
|---|---|---|
| 孩子接受度 / 用户接受度 | **目标受众分层(Audience Stratification)** | 营销 / 品牌 |
| 决策端 vs 使用端 | **利益相关者 vs 终端用户(Stakeholder vs End User)** | 服务设计 |
| 在算交付链 | **交付链路前置规划(Critical Path Pre-assessment)** | 项目管理 |
| 直接出图 | **可视化方案先行(Visual-First Proposal)** | 设计 |
| 看图落地 | **应用场景样机(Contextual Mockup)** | UX |
| 心中的筛子 | **视觉本能识别(Visual Pattern Recognition)** | 认知科学 |
| 对下游友好 | **全链路设计交付意识(End-to-End Delivery Awareness)** | 服务设计 |
| 懂物料制作 | **生产可行性预判(Production-readiness Foresight)** | 设计制作 |
| 亲和力 | **视觉亲和力(Visual Affinity)** | 设计 |
| 脑暴 / 头脑风暴 | **前期发散阶段(Divergent Phase)** | 设计思维 |
| 反复改 | **迭代审议(Iterative Review)** | 项目管理 |
| LOGO 三层 | **三层结构整合(Compositional Layering)** | 品牌 |
| 母 LOGO 衍生 | **提示词推演衍生(Prompt Lateral Derivation)** | AI 设计 |

## 不要换的词(口气词保留)

> 这些词带着用户自己的味道,**不要包装成 corporate 术语**:

- "兜底能力"(用户原创,工艺级感)→ **不要换成"全链路 / 闭环交付"**
- "一眼卯定"(用户原创,直觉感)→ 不要换
- "鹿火车只是元素清单"(用户场景化表达)→ 不要换
- "因为我干过下游的工作"(经历感)→ 不要换
- "12 小时一夜爆三方案"(节奏感)→ 不要换

## 反向自检 5 项(每段写完前过)

| # | 检查项 | 标准 |
|---|---|---|
| 1 | **服务视角** | 这一段的主语是"我"还是"项目方"?有没有服务视角注脚? |
| 2 | **专业度** | 主文里有没有大白话直接出现?对照替换表 |
| 3 | **保留口气** | 用户的口气词("兜底能力"等)有没有被换成 corporate 术语? |
| 4 | **金句锚点** | 有没有一句中英对仗术语作为整段钩子? |
| 5 | **素材闭环** | 每条主张有 1-3 类佐证(数据 / 引用 / 图)? |

5 项全过 → 这一段可以交付。
