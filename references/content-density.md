# 段落级信息密度规则(Content Density)

每段 `<section class="section">` 必须配齐的字段 + 内容来源 + 长度 + 不能写什么。

> **核心原则**:招聘方一个人扫,看完就 get,不需要任何外部解释。
> 每段 = "完整可阅读的故事单元"(类比一本书的小节,不是一张幻灯片的标题)。

---

## 必配 9 件 ✅

每段 `.section` 必有以下 6 件(P0 必过)+ 按需 3 件:

| # | 字段 | 必有 | 长度 | 用途 |
|---|---|---|---|---|
| 1 | `.chrome`(段顶元数据) | ✅ | 一行 | 段号 / 章节归属 / 英文术语 |
| 2 | `.kicker`(短前缀) | ✅ | 一行 | 段标签 |
| 3 | `<h3>` 主标 | ✅ | 一句(< 25 字) | 这一段的"答案" |
| 4 | `.h-sub` 副标 | ✅ | 一句(< 30 字) | 主标的展开 |
| 5 | `<p class="body">` 副文 | ✅ | 2-3 段(每段 80-150 字) | 完整论述 |
| 6 | `.service-note`(服务视角注脚) | ✅ | 一段(40-80 字) | 项目方价值 |
| 7 | `.anchor-quote`(锚点金句) | 推荐 | 1 句(< 30 字) | 整段视觉钩子 |
| 8 | `.callout`(原话引用) | 推荐 | 1-2 条 | 佐证脚注 |
| 9 | 数据卡 / 对比 / Pipeline / 图 | 按需 | — | 视布局而定 |

---

## 字段详解

### 1. `.chrome`(段顶元数据)

**写什么**:段号 + 4 字标签 + 英文术语,右侧是章节归属 + 段顺序。

```html
<div class="chrome">
  <div>1A · 项目定位 / Project Footprint</div>
  <div>Act I · 02</div>
</div>
```

**长度**:每行 < 40 字符。
**反例**:不要写"这是 1A 段,这一段讲项目定位..."(口语化,跟章节标题重复)。
**正确**:简洁的"段号 · 标签 / 术语" 对仗式。

### 2. `.kicker`(短前缀)

**写什么**:跟 `.chrome` 左侧呼应的"段号 · 4 字标签"。

```html
<div class="kicker fade-in">1A · 项目定位</div>
```

**长度**:< 15 字。
**作用**:大标题前的"小前缀",让招聘方滚到这一段时**先看到一个小标签**,再看大标题。

### 3. `<h3>` 主标(关键!)

**写什么**:**这一段要让招聘方相信的一句话**。

```html
<h3 class="fade-in">7 天 · 一个人 · 一套品牌全案。</h3>
```

**长度**:< 25 字(中文)。
**写法**:
- 用**句号结尾**(陈述句,不是疑问句)
- 数字 + 关键词 + 关键词的对仗节奏(如"利益相关者 ≠ 终端用户")
- **不能**用大白话(如"我做的项目挺好的")
- **不能**用术语堆砌(如"基于 toC 公益设计场景的目标受众分层结构性识别")—— 太学究
- **关键平衡**:**专业术语骨架 + 口语化节奏**

**反向自检**:
- 这句话**单独看**(脱离上下文),招聘方读完会有什么反应?
- 如果反应是"嗯,这人在讲什么具体的事" → 留
- 如果反应是"这句话听上去像 AI 写的 / 像万金油" → 改

### 4. `.h-sub` 副标

**写什么**:主标的**展开半句**,用比主标低一档的字号。

```html
<p class="h-sub fade-in">从接手到交付的硬指标。</p>
```

**长度**:< 30 字。
**作用**:让招聘方扫主标 → 扫副标 → 决定要不要读副文。
**不能**:跟主标重复(如主标"7 天 · 一个人",副标"7 天交付一个人扛")。

### 5. `<p class="body">` 副文

**写什么**:这一段的**完整论述**。**所有大白话用 Step 7 词汇升级表替换**。

```html
<p class="body fade-in">
  在 toC 公益项目里,**利益相关者**(决策端)和 **终端用户**(使用端)
  是结构性分离的——这不是某个项目的特殊情况,而是 toC 设计的常态。
</p>
```

**长度**:每段 80-150 字。**2-3 段为标准**(< 5 段)。
**强调元素**:
- `<strong>` 重点术语(中文)
- `<em>` 英文术语(自动 italic)
- `<br>` 不用(段间用 `<p>` 分)

**写法**:
- 第 1 段:点明本段要解决的问题
- 第 2 段:你怎么解决的(方法 + 关键决策)
- 第 3 段(可选):为什么这么解决比其他方案好

**反例**:
- ❌ "项目方很急,我帮他们做完了" —— 大白话
- ❌ "基于 toC 公益场景的目标受众分层维度的设计判断输出" —— 学究
- ❌ "我用 4 款 AI 产品在 12 小时内做了 15 个并行方案" —— 自我表现(应该是项目方拿到什么)

**正确**:
- ✅ "项目方提供的「鹿、火、车」是**视觉元素清单**,服务于决策端的审稿口味;LOGO 真正落地面对的是终端用户——孩子的眼睛。两端对「好 LOGO」的判断标准,*结构性地不重合*。"

### 6. `.service-note`(服务视角注脚)

**写什么**:**招聘方读完这一段会理解什么** —— 项目方价值的明示。

```html
<div class="service-note fade-in">
  <span class="label">服务视角</span>
  招聘方读到的是:这人在 toC 项目的第一秒就在做「目标受众分层」 —— 不被项目方需求清单牵着走,会先识别真正的服务对象。这是有用户研究底子的品牌设计师才有的判断顺序。
</div>
```

**长度**:40-80 字。
**写法**:
- 必须用"**招聘方读到的是**"或"**招聘方 get 到的是**"开头
- 接一段对项目方价值的概括
- 不能用第一人称"我"

**作用**:这是**质量门 P0** —— 没这个 = 没过门。

### 7. `.anchor-quote`(锚点金句)

**写什么**:整段的**视觉钩子** + **认知钩子**,中英对仗。

```html
<div class="anchor-quote fade-in">
  视觉亲和力(Visual Affinity)<br>
  是项目战略变量,不是审美偏好。
</div>
```

**长度**:< 30 字。
**写法**:
- 中英对仗(英文术语 + 中文展开)
- 句子有节奏(可断行)
- 不直接重复主标
- **每段 1 句** —— 多了就没钩子感

**经典模式**:
- "Show, don't tell — 让审议端从「读方案」变成「看方案」"
- "本能先响,逻辑跟上 — 内化到位的方法论,长成视觉本能"
- "Compositional Layering — 把 LOGO 拆成可独立设计、可整合迭代的三层结构"

### 8. `.callout`(原话引用)

**写什么**:用户原话 / Get 笔记 / 复盘文档的**直接引用**。

```html
<div class="callout fade-in">
  "如果 logo 很机械、很冷酷,孩子看到后会觉得这个项目很陌生,甚至有距离感。"
  <span class="src">— Get 笔记 2025-12-14</span>
</div>
```

**长度**:每条 < 80 字。每段 1-2 条。
**用途**:**佐证脚注层** —— 不能取代副文。

**写法**:
- 引号包裹(中文双引号 " ")
- `.src` 来源:用户名 / 文档名 / 时间(等宽小字)
- **保留用户口气词**(如"兜底能力""一眼卯定")—— 这就是 callout 的最大价值

**反例**:
- ❌ 引用过长(超 100 字) —— 长引用断成两条
- ❌ 引用脱离原意 —— 必须忠实于用户原话

### 9. 数据卡 / 对比 / Pipeline / 图(按需)

**视布局而定**(见 sections.md):
- 数据大字报(Section 3) → `.stats-grid` 6 卡 / 3 大卡
- 左文右图(Section 4) → `.split` 7:5
- Before/After(Section 5) → `.compare` 1:1
- Pipeline(Section 6) → `.pipeline-block` 双流水线
- 图片网格(Section 7) → `.family-grid` 3×2

## 反向自检 5 项(每段写完前必过)

| # | 检查项 | 标准 |
|---|---|---|
| 1 | **服务视角** | 这一段的主语是"我"还是"项目方"?有没有 `.service-note`? |
| 2 | **专业度** | 主文里有没有大白话直接出现?对照术语库替换表 |
| 3 | **保留口气** | 用户的口气词("兜底能力"等)有没有被换成 corporate 术语? |
| 4 | **金句锚点** | 有没有 `.anchor-quote` 一句中英对仗术语? |
| 5 | **素材闭环** | 每条主张有 1-3 类佐证(数据 / 引用 / 图)? |

**5 项全过 → 这一段可以交付**。
**任意一项不过 → 不要交,改完再过门**。

## 大白话 → 专业术语 替换表

| 大白话 | 专业术语 | 来源 |
|---|---|---|
| 孩子接受度 / 用户接受度 | **目标受众分层(Audience Stratification)** | 营销/品牌 |
| 决策端 vs 使用端 | **利益相关者 vs 终端用户(Stakeholder vs End User)** | 服务设计 |
| 在算交付链 | **交付链路前置规划(Critical Path Pre-assessment)** | 项目管理 |
| 直接出图 | **可视化方案先行(Visual-First Proposal)** | 设计 |
| 看图落地 | **应用场景样机(Contextual Mockup)** | UX |
| 心中的筛子 | **视觉本能识别(Visual Pattern Recognition)** | 认知科学 |
| 对下游友好 | **全链路设计交付意识(End-to-End Delivery Awareness)** | 服务设计 |
| 懂物料制作 | **生产可行性预判(Production-readiness Foresight)** | 设计制作 |
| 亲和力 | **视觉亲和力(Visual Affinity)** | 设计 |
| 脑暴 / 头脑风暴 | **前期发散阶段(Divergent Phase)** | 设计思维 |
| 反复改 | **迭代审议(Iterative Review)** | 项目管理 |
| 抽象判断 vs 具象判断 | 同字面 | 认知 |
| LOGO 三层 | **三层结构整合(Compositional Layering)** | 品牌 |
| 母 LOGO 衍生 | **提示词推演衍生(Prompt Lateral Derivation)** | AI 设计 |

## 不要换的词(口气词保留)

> 这些词带着用户自己的味道,**不要包装成 corporate 术语**:

- "兜底能力"(用户原创,工艺级感)→ **不要换成"全链路 / 闭环交付"**
- "一眼卯定"(用户原创,直觉感)→ 不要换
- "鹿火车只是元素清单"(用户场景化表达)→ 不要换
- "因为我干过下游的工作"(经历感)→ 不要换
- "12 小时一夜爆三方案"(节奏感)→ 不要换
- "20cm 拍照效果不好"(具体观察)→ 不要换
